## 这是什么？
这个游戏的核心是“事件驱动”。玩家将在人生中不停地经历各种事件，你写一个事件 → 游戏加载它 → 玩家遇到它 → 做出选择 → 触发结果。如果您也有意愿参与创作甚至是分享您的事件，请允许我对您介绍一下标准的实现方式。  
当然，《逗比人生》另外一个核心元素是“特质”，得益于python的便利性，特质不需要显式定义，您只需要在和事件交互中提及任何特质，游戏便会自动的处理它。

## 事件是什么？
首先，请允许我介绍一下本游戏的事件处理逻辑。游戏内与事件相关的容器总共有三个，分别是**事件库**、**事件池**和**事件队列**，在游戏中这三个容器一并生效。   
所有的事件（包括您即将编写的）都会在游戏开始时加载进**事件库**，这是存放所有事件的基本容器。你可能会经历的事件是存放在**事件池**内的。游戏会在进程中不断地扫描**事件库**，将满足条件的事件加入**事件池**，并将不满足条件的事件从池中放回库中，然后，系统会从**事件池**中带权随机地取出一个事件，作为玩家正在经历的事件，这个过程会不断循环。    
而**事件队列**是为了实现子事件的处理，这里不多赘述了。
对于任何一个事件都有若干个选项，玩家会选择一项，然后这个选项会根据玩家的情况导向一个结果，之后本事件结束，游戏发展到下一个事件。    
因此，对于一个基本事件，您需要描述它的基本部分包括相关的文本，存在的选项，事件加入池中的条件三部分。而游戏内对事件的基本管理需要这个事件的名字，因此必须包含的部分就是事件名、显示文本、包含选项和条件四部分。如果您想要某个事件导向另一个事件，可选项包括事件的子事件名。

## 事件的基本格式
```json5
{
  "事件名": {
    "name": "事件名",
    "title": "显示标题",
    "text": "事件描述文本",
    "conditions": {},       // 什么条件下才会加入池中 条件的格式请参照下文
    "weight": 256,          // 出现概率（权重）
    "options": [            // 玩家可以做的选择
      {
        "text": "选项文字",
        "conditions": {},
        "result": {
          "branches": [
            {
              "conditions": {},
              "result_str": "选择后的结果描述",
              "effect": {}   // 实际效果（加特质、推子事件等）
            }
          ]
        }
      }
    ]
  }
}
```
---
##  选项又怎么写？
选项是玩家在经历事件的可选择对象。一个事件应该至少包含一个可选选项。当玩家选择这个选项时，根据玩家的不同状态，通常也会有不同的效果，这就是**结果分支**，也是您需要编写的部分。
> 例如，当玩家选择游过河的选项时，如果玩家有“会游泳”的特质，那么玩家就可以安全过河，否则会淹死，这就是结果分支的存在价值  

选项同样拥有自己的条件，这个条件决定了这个选项是否可选。
```json5
"options": [            // 玩家可以做的选择
      {
        "text": "选项文字",
        "conditions": {}, //这个选项可以被选择的条件
        "result": {
          "branches": [ //所有的结果分支
            {
              "conditions": {}, //每个分支的条件
              "result_str": "选择后的结果描述",
              "effect": {}   // 实际效果（加特质、推子事件等）
            }
          ]
        }
      }
    ]
```

### 关于结果分支 
游戏会从上到下地查看所有结果分支，并找到第一个条件满足的结果分支，应用其结果描述和实际效果。
实际效果的全部字段实现：

| 字段                      | 效果                             |
|:------------------------|:-------------------------------|
| "add_traits":[]         | 为玩家添加[ ]内的所有特质                 |
| "remove_traits":[]      | 为玩家删去[ ]内的所有特质                 |
| "age_up":x              | 让玩家增长x岁                        |
| "sub_event":"next_name" | 增加名为"next_name"的子事件，下一事件即为此子事件 |

## 条件怎么用？
条件可以放在三个地方：

 | 位置  | 作用             |
 |:----|:---------------|
 | 事件上 | 控制“这个事件会不会被抽到” |
 | 选项上 | 控制“这个选项会不会显示”  |
 | 分支上 | 控制“这个结果会不会生效”  |


 |       字段        |           含义            |
 |:---------------:|:-----------------------:|
 |  age_limit_min  |          	最小年龄          |
 |  age_limit_max  |          	最大年龄          |
 |   need_traits   |        	必须拥有这些特质        |
 | no_need_traits	 |        不能拥有这些特质         |
 |    need_sx	     |      性别要求（0=男，1=女）      |
 |     is_son	     |   是否为子事件，普通事件不要包含本条目    |
|   probability   | 概率发生，这个条目写百分比下发生的概率，正整数 |

您不必在写条件时将上述条目全部加入，只需要加入您需要的即可。例如，您需要判断最小年龄，但不要求最大年龄，可以只添加`"age_limit_min"`字段。

## 一个完整示例
```json5
{
  "示例事件1": {//本处使用中文出于方便理解，实际使用时建议您改用英文
    "name": "示例事件1",//同上，游戏内实际用于标识事件的标识符为本条目，建议您和上方的字符串完全相同
    "title": "这是本事件的标题，会用黄色粗体显示在事件最上方",
    "text": "这是本事件的文本，会显示在标题下方，对整个事件做描述",//您可以使用制表符来完成换行等行为
    "conditions": { //条件字典适用于事件、选项和分支，分别决定事件会不会被抽到、选项会不会出现、是否实现这个结果分支，下述列出了所有可用的条件，您在创作时不需要全部加入，不需要判断的条件就无需写对应条目
      "age_limit_min": 0, //最低年龄限制，规定满足条件的最低玩家年龄，非出生的事件建议您选择大于0的数字，避免玩家不能正常出生
      "age_limit_max": 100,//最高年龄限制，规定满足条件的最高玩家年龄
      "need_traits": ["需要特质1","需要特质2"],//这个字符串列表规定了所有满足条件所需要的特质，目前不支持或逻辑，必须全部满足，您可以增加更多或减少
      "no_need_traits": ["不能有的特质1","不能有的特质2"],//这个字符串列表规定了所有满足条件所不能有的特质
      "need_sx": 0, //规定玩家触发的性别，0为男，1为女，暂时只支持二元性别，不代表本人歧视性别多样性，也不代表本人支持性别多样性，仅为技术便于实现选择
      "is_son": 0 //表示是否是子事件的判断，仅用在事件内的条件，如果是子事件，那么这个事件不会自然被抽取到，仅会通过其他事件的子事件传入
    },
    "weight": 256,//由于事件是权重制抽取，这一栏的数字表示了这个事件有多大概率会被抽到，建议将256作为基准值，您可以在此基础上上调或下调
    "options": [//这里是选项，您至少需要写一个选项，且最好保证至少有一个选项总是可选，您可以添加任意多个选项，只要格式相同
      {
        "text": "这是选项1对应的文字",
        "conditions": {},//这是选项出现在可选列的条件，格式与上面的conditions相同，如您所见，条件是可以空置的
        "result": { //即便选择了某个选项，也有可能因为不同的特质组合而有不同的结果，result和下面的branches正是用来实现这一点
          "branches": [
            {
              "conditions": {
                "need_traits": ["本分支所需特质"]
              },
              //游戏会按您的顺序遍历所有可能的branch，并找到第一个满足条件的branch，并将下方的值返回给结果
              "result_str": "满足这个分支下，会显示的结果文本",
              "effect": {
                //effect是一种表示实现效果的字典，下述列出了所有目前可以实现的效果，同样的，您不需要全部添加每个条目
                "add_traits": [
                  "要加入的特质1",
                  "要加入的特质2"
                ],
                //为玩家添加某些特质
                "remove_traits": [
                  "要删除的特质1",
                  "要删除的特质2"
                ],
                //为玩家删去某些特质
                "age_up": 1,
                //增加玩家的岁数，正整数，当然您也可以输入负整数，但文本表现可能会比较奇怪
                "sub_event": "子事件name"
                //添加一个子事件到队列首，这意味着下一个事件一定是子事件name指向的事件
              }
            },
            {
              "conditions":{}, 
              "result_str": "另一个分支",
              "effect": {}
            }
          ]
        }
      },
      {
        "text": "这是选项2对应的文字",
        "conditions": {},
        "result": {
          "branches": [
            {
              "conditions": {},
              "result_str": "这是这个分支对应的结果文本。",
              "effect": {
              }
            }
          ]
        }
      },
    ],
    "sub_event": "子事件name"//指示了这个事件固定会派生的子事件名，这是可选项，如果没有子事件，您可以不加这一栏
  }
}
```

## 怎么测试我的事件？
请在`main/data/packages/`下创建您的包文件夹，将您编写的事件文件`events.json5`复制至该文件夹下，同时，在该文件夹下创建文件`package.yml`，复制以下内容并改为您自己的内容：
```yml
name: "示例事件包"
# 您可以用任意的名称，但id请尽量使用独一无二的，建议可以使用下划线规范化命名
id: "example_name"
# 这是游戏搜寻您的包时用的包名，请使用英文，可以自定义
version: "0.0.0"
# 版本号由您自己决定
events_file: "events.json5"
# 建议本栏保持不变，您可以选择json或是json5后缀，本示例文件为了方便json文件的注释，选用了json5格式，您如果成品文件不包含注释或非标准行为，可以换回json
```
然后编辑`main/data/config.yml`，在`packages`字段的最下方，增加一条：
```yml
    - 您的事件包id
```
之后运行游戏，应该可以看到符合条件的事件被正确加入了游戏。
